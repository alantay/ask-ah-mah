# ADR-0013 — Market Tips are LLM-generated and shared, not curated or per-user

**Status:** Accepted

## Context

A new feature gives the user **Market Tips** — Ah Mah's advice on how to *pick* a fresh item well at the shop ("firm, deep-red tomato, no bruises"; "a dark avocado that gives slightly is ripe today"). Tips attach to the missing items (**Additions**) on a recipe's shopping list.

Two questions had to be answered:

1. **Where do tips come from and where are they stored** — a per-user table, a hand-curated shared table, or generated by the model?
2. **Why is a freshness-adjacent feature acceptable at all**, given [ADR-0008](0008-no-shelf-life-ui.md) deliberately removed the shelf-life / "use soon" system?

A future reviewer will reasonably ask both — especially the second, since "freshness" was explicitly killed once already.

## Decision

### Tips are LLM-generated, then cached in a shared, item-keyed table

The first time any user needs a tip for an item, Ah Mah generates it; it is persisted in a single shared corpus keyed by a **canonical item name** (e.g. recipe ingredient *"2 ripe tomatoes, diced"* normalizes to `tomato`). Every subsequent request — for any user — reads from the cache. The table has **no `userId`**.

Only *fresh / pickable* items (produce, fruit, seafood, meat) get a tip. For staples (salt, sauces, dry goods) the model returns "none": nothing is cached and no tip affordance renders.

### Tips surface on the recipe shopping list

A tip is revealed inline per missing item on the shortfall card, and folded into the "Copy shopping list" output so it travels to WhatsApp / Notes — the surface the user actually reads while shopping. No standalone persisted shopping list is built in this iteration.

## Why this clears ADR-0008

ADR-0008 rejected shelf-life because **freshness of a specific item cannot be reliably known from app state** — `dateAdded` is not acquisition or expiry date, and steering recipes by guessed urgency degrades them.

A Market Tip asks a *different* question. Shelf-life: "is *my* tomato still good?" — per-user, time-dependent, unknowable. A Market Tip: "how do I pick a good tomato?" — universal, time-independent, general knowledge the model already holds. Nothing about a Market Tip depends on app-state accuracy, so the failure mode that sank shelf-life does not apply.

## Why not the alternatives

**Per-user tips.** Picking advice is a universal fact, identical for every user; storing it per account multiplies identical rows and offers no personalization payoff. Justified only if the value were user-*authored* notes — which it is not.

**Hand-curated shared table.** Deterministic and editorially controlled, but someone must author a tip for every conceivable item; coverage is capped at what was written, and gaps are silent. The model already knows this material — curation spends human effort to reproduce it.

**Pure LLM with no cache.** Simplest data model, but every list re-pays latency and cost for advice that never changes, and offers nothing offline. Caching universal, unchanging facts is strictly better; the shared cache also grows a reusable corpus for free.

## Consequences

- New shared model (e.g. `MarketTip { key @id, tip, createdAt }`) with **no `userId`**.
- Generation must normalize ingredient strings to a canonical key, or the cache will not hit.
- The model contract must support returning "no tip" for staples, so the cache and UI stay free of filler.
- ~~The shortfall card's ≥50%-in-pantry gate is narrow; tips should attach wherever missing ingredients already render (verify render paths during planning).~~ **Resolved (Jun 2026):** the shortfall card was conflating two jobs — encouragement ("You're almost there") and utility (shopping list + tips). The ≥50% ratio now only switches the *heading copy*; the card itself (and its tips) renders whenever the user owns ≥1 ingredient and is missing ≥1. See the "Shortfall card" glossary entry in `CONTEXT.md`.
- A standalone, persisted shopping list remains deferred — this iteration proves the tip value on the existing recipe-bound list first.
